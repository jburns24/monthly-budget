# Research Report: Categories Management (Epic 4)

> Generated by cw-research on 2026-04-04
> Codebase: monthly-budget

## Summary

- **Epic 4 (Categories Management)** is a CRUD feature for flat, family-scoped spending categories with soft-delete/archive semantics. It depends on Epic 3 (Family Management) and is depended on by Epics 5 (Expenses) and 6 (Goals). Monthly Goals are **out of scope** (Epic 6).
- **No category code exists yet** — model, migration, service, router, schemas, frontend API, types, components, and tests all need to be created from scratch. The BottomNavigation already has a disabled "Categories" tab placeholder.
- The codebase follows a **consistent layered architecture**: SQLAlchemy 2.0 async models → service functions (raise HTTPException, use `db.flush()`) → FastAPI routers with Depends-based RBAC → Pydantic schemas. Frontend mirrors this with `apiClient()` → React Query hooks → Chakra UI v3 components with role-based conditional rendering.
- **RBAC is well-established**: `require_family_member` (read) and `require_family_admin` (write) FastAPI dependencies return `(User, FamilyMember)` tuples. Categories must use admin-only writes, member reads — identical to existing family management patterns.
- **Testing is comprehensive**: Backend uses pytest-asyncio with NullPool per-test rollback, factory functions, and JWT patching. Frontend uses Vitest + Testing Library with mocked API modules. E2E uses Playwright with dev-auth bypass, Page Objects, and `resetTestData()` isolation.

## Architecture & Patterns

### Design Patterns

**Service Layer (async functions, not classes)**
> See: `backend/app/services/family_service.py`

- Functions receive `db: AsyncSession` + business parameters
- Use `await db.flush()` (never `commit()`) — transaction committed by `get_db()` dependency
- Raise `HTTPException` directly with appropriate status codes
- Log with `structlog.get_logger(__name__)` and structured key-value pairs

**Router Layer**
> See: `backend/app/routers/family.py`

- `APIRouter(prefix="/api", tags=["families"])`
- Dependencies: `Depends(require_family_member)` for reads, `Depends(require_family_admin)` for writes
- Both return `tuple[User, FamilyMember]`; routers unpack as `current_user, _ = membership`
- Manual ORM-to-Pydantic conversion via `_model_to_response()` helper functions
- Response models declared in decorator: `response_model=FamilyResponse`

**Frontend Component Pattern**
> See: `frontend/src/components/family/CreateFamilyView.tsx`

- Controlled inputs with `useState`
- `useMutation` from React Query with `onSuccess` (toast + invalidate queries) and `onError` (toast)
- `useFamilyContext()` for familyId/role; `useAuth()` for current user
- Conditional rendering: `{isAdmin && <AdminOnlyComponent />}`
- Dialog pattern: `DialogRoot` with `open`/`onOpenChange` controlled by `useState<TargetType | null>`

### Module Boundaries

- **Backend**: Models → Services → Routers → Schemas (strict layering)
- **Frontend**: API functions → React Query hooks (in components) → UI components
- **Cross-cutting**: `FamilyContext` provides `familyId`/`role` to all protected pages; `apiClient()` handles auth token refresh transparently

### Key Abstractions

- `get_db()` — Async generator dependency: yields session, commits on success, rollbacks on exception
- `require_family_member` / `require_family_admin` — RBAC dependencies returning `(User, FamilyMember)` tuple
- `apiClient()` — Fetch wrapper with automatic 401 → refresh → retry queue pattern
- `FamilyProvider` / `useFamilyContext()` — Derives familyId/role from `useAuth().user.family`

### Conventions

| Area | Convention |
|------|-----------|
| Error privacy | 404 for non-members (not 403) to avoid leaking family existence |
| Error detail | Dev: specific message; Prod: generic "Authentication required" |
| Timestamps | `TIMESTAMP(timezone=True)`, UTC stored, ISO 8601 serialized |
| IDs | UUID everywhere (no sequential IDs) |
| Naming | snake_case (Python), camelCase (TypeScript), kebab-case (URLs) |
| Logging | Structured: `logger.info("event_name", key=value)` |
| Toasts | `toaster.create({ title, description, type, duration: 4000 })` |

### Categories Schema (from PRD §10)

```sql
CREATE TABLE categories (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    family_id       UUID NOT NULL REFERENCES families(id) ON DELETE CASCADE,
    name            VARCHAR(100) NOT NULL,
    icon            VARCHAR(50),
    sort_order      INTEGER NOT NULL DEFAULT 0,
    is_active       BOOLEAN NOT NULL DEFAULT true,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (family_id, name)
);
CREATE INDEX idx_categories_family ON categories(family_id);
```

**Key design decisions**:
- `is_active` (not `deleted_at`) for soft-delete/archive
- `ON DELETE CASCADE` from families (family deletion removes all categories)
- `ON DELETE RESTRICT` on expenses.category_id (prevents hard-delete when expenses exist)
- `UNIQUE (family_id, name)` prevents duplicate names within a family
- Flat structure (no parent_category_id) — hierarchy is out of scope

## Deep-Dive Findings

### Backend Architecture Details

**Transaction management**: All services use `await db.flush()` (not commit). The `get_db()` dependency commits after the router returns successfully, or rollbacks on exception. This allows multiple service calls within a single transaction.

**Eager loading**: Family queries use `joinedload()` chains with `.unique()` to avoid N+1 problems. Categories will need this when returning related data.

**No optimistic locking on categories**: The PRD specifies `version` field only on `monthly_goals` (Epic 6). Categories use simple CRUD without concurrency control.

**Error handling convention**: Services raise `HTTPException` directly. Privacy-preserving: 404 for non-members, 403 for insufficient role. The `_auth_error()` helper in dependencies.py returns generic messages in production.

### Frontend Architecture Details

**BottomNavigation** already has a disabled "Categories" tab at position 2 (between Home and Family) with a 4-square grid icon. Implementation requires: enabling the tab, adding `/categories` route to App.tsx, creating CategoriesPage component.

**Provider hierarchy**: `ChakraProvider → QueryClientProvider → BrowserRouter → App → (ProtectedLayout: FamilyProvider + BottomNavigation)`. Categories page will be inside ProtectedLayout, accessing FamilyContext.

**Form pattern**: Controlled inputs with `useState`, `useMutation` with closure over state, `toaster.create()` for feedback, `queryClient.invalidateQueries()` on success. Button shows `loading={mutation.isPending}`, disabled when input empty.

**Role-based rendering**: `const isAdmin = role === 'admin'` from `useFamilyContext()`. Admin-only actions wrapped in `{isAdmin && <Component />}`.

### PRD Requirements Summary

**Suggested default categories** (onboarding flow, PRD §6.2 line 190): Groceries, Dining, Transport, Entertainment, Bills, Other. Note: onboarding is a **separate epic** per decision in `00-epic-breakdown-questions-1.md`, but Epic 4 must support the CRUD that onboarding will use.

**Archive behavior** (PRD §5 line 144): "Category is archived (hidden from forms, visible in history)." Archived categories must not appear in category dropdowns but must still display on historical expense records.

**Category suggestion** (PRD §13 line 875): pg_trgm trigram matching on expense descriptions (similarity > 0.6). This is used by the receipt pipeline (Epic 7) but the pg_trgm extension and index setup may be included in Epic 4's migration.

**Gaps identified**: No wireframe for the categories management page. No explicit request/response body schemas in PRD. No maximum categories per family specified. Icon format unspecified (emoji vs icon code).

### Testing Infrastructure Details

**Backend test factory** to add: `create_test_category(db, family, **overrides)` following the pattern of `create_test_family()`. Returns `Category` ORM object.

**Test reset endpoint** (`/api/test/reset`) will need updating to truncate categories table in the correct FK order (before families, after expenses if they exist).

**E2E page object** to create: `CategoriesPage` following `FamilyPage` pattern with locators for create button, name input, category list items, edit/delete actions.
